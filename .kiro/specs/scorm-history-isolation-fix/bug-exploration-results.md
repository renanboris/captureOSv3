## Root Cause Analysis

All test failures confirm the hypothesized root cause from the design document:

### Bug Location
**File**: `scorm_eng/templates/js/try-player.js`
**Line**: 19

### Problematic Code
```javascript
window.moduloId = urlParams.get('modulo') || 'default';
```

### Why It's a Bug
1. This line executes immediately when the script loads, BEFORE `STEPS_DATA` is loaded
2. In standalone mode without URL `modulo` parameter, it always sets `moduloId = 'default'`
3. The `session_id` from `STEPS_DATA` is available later (loaded at line 28-35) but never used
4. All SCORMs end up with the same `moduloId = 'default'`
5. The `scorm-api.js` uses `moduloId` to generate localStorage keys: `scorm_${moduloId}_${key}`
6. Result: All SCORMs share keys like `scorm_default_cmi.core.lesson_status`

### Timing Issue
```
Current Execution Order (BUGGY):
  Line 19: window.moduloId = 'default'  ← Set too early
  Line 28-35: Load STEPS_DATA with unique session_id  ← Too late
  Line 53: ScormAPI.get() uses moduloId='default'  ← Uses wrong value

Needed Execution Order (FIXED):
  Line 28-35: Load STEPS_DATA with unique session_id
  After 35: window.moduloId = STEPS_DATA.session_id  ← Use unique ID
  Line 53: ScormAPI.get() uses moduloId=session_id  ← Uses correct value
```

## Bug Condition

The bug manifests when ALL of the following conditions are true:

```javascript
isBugCondition(input) {
    return (NOT input.isLMS) AND
           (input.urlParams.get('modulo') IS NULL) AND
           (input.moduloId = 'default')
}
```

**In plain language**: SCORM runs in standalone mode (not in an LMS) AND there's no `?modulo=xyz` parameter in the URL AND the code falls back to `moduloId='default'`.

## Impact Assessment

### Severity: HIGH

**User Impact**:
- Users completing SCORM_A and then opening SCORM_B see SCORM_B as already completed
- Progress from one training incorrectly appears in a completely different training
- Users may skip required training thinking they already completed it
- XP scores, lesson locations, and completion status leak between unrelated SCORMs

**Data Integrity**:
- Progress data (`cmi.suspend_data`) from one SCORM overwrites another
- Lesson status (`cmi.core.lesson_status`) incorrectly shared
- Score data (`cmi.core.score.raw`) contaminated across SCORMs

**Scope**:
- Affects ALL standalone SCORM executions without URL `modulo` parameter
- Does NOT affect LMS-hosted SCORMs (they use API, not localStorage)
- Does NOT affect SCORMs with explicit `?modulo=xyz` URL parameter
